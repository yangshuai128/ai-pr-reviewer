# 架构与设计说明

本文件回答题目要求说明的三点（**模型选择 / 上下文获取方式 / 未来扩展方向**），
并说明对五个关键因素（准确性、上下文理解、误报与漏报控制、响应速度、使用体验）的设计思路。
每个决策按「业界做法 → 我的选择 → 取舍理由 → 未来方向」四段式展开。

---

## 一、整体架构

```
PR 链接
  → github_client（取数：调 GitHub API 拿 diff）
  → context_builder（组织上下文：简化版 PR 压缩）
  → 并行：
      ├── analyzer（LLM：Claude 出语义问题）
      └── static_analyzer（ruff：出确定性问题）
  → pipeline（合并两路结果）
  → reporter / app（命令行报告 / 网页展示）

eval/ 评估脚本：用已知问题量化召回率
```

模块解耦：取数、上下文、分析、展示各自独立，便于单独测试与未来替换。

---

## 二、模型选择

**业界做法**：PR-Agent 强调“每个工具一次 LLM 调用”，追求快（约 30 秒）且便宜；
CodeRabbit 则用更重的多步推理换取更高准确率。

**我的选择**：使用 `claude-sonnet-4-6`。它在准确性与速度/成本之间平衡较好
（约 $3/$15 每百万 token）。代码中预留了 `claude-haiku-4-5-20251001` 作为
高并发、成本敏感场景的更快更省选项，一行即可切换。

**取舍理由**：实训项目需要在有限预算内反复测试，Sonnet 的性价比最合适；
不选 Opus 是因为评审任务对成本敏感，Opus 的额外推理力对本任务收益有限。

**未来方向**：按 PR 规模/语言做模型路由——小 PR 用 Haiku，复杂架构 PR 用 Opus。

## 三、上下文获取方式

**业界做法**：PR-Agent 的核心能力是 “PR Compression”，把任意大小的 diff 压成
可控大小的提示词；CodeRabbit 做“上下文工程”，构建代码依赖图谱，并把约 80–90%
的 token 预算花在上下文构建而非最终评审。

**我的选择**：实现了**简化版 PR 压缩**：
1. 按“信息价值”给文件排序（源码 > 测试/配置；改动行数越多越优先）；
2. 在总字符预算（默认 15000）内分配；
3. 超预算时只保留 diff 的变化行（`+`/`-`/`@@`），丢弃未改动的上下文行；
4. 逐文件分析，天然支持大 PR 分批。

**取舍理由**：相比“粗暴截断尾部”，按价值排序 + 只留变化行能在同样预算内
保住最关键的改动，避免把重要代码切掉；又远比构建完整代码图谱省时，契合两天工期。

**未来方向**：引入 AST/代码依赖图谱，把被改函数的完整定义与调用方一并纳入上下文，
向 CodeRabbit 的上下文工程靠拢。

## 四、误报与漏报控制（准确性核心）

**业界做法**：CodeRabbit 把 LLM 与 40+ 个 linter/SAST 工具结合；Kodus 用 AST+LLM
混合架构降低幻觉。共识是：纯 LLM 评审噪音多、易幻觉，叠加确定性工具能显著提升准确率。

**我的选择**：**混合架构**——
- 确定性问题（语法、未用变量、明显风格）交给 `ruff`，零误报；
- 语义性问题（逻辑 bug、设计问题）交给 Claude；
- 两路结果分栏展示，互不污染。
- 同时让模型为每条风险标注**置信度**，便于按置信度过滤，进一步压低误报。

**量化**：`eval/run_eval.py` 用人工标注的已知问题统计召回率，
为“漏报控制”提供可度量的依据；精确率可在此基础上人工标注误报后计算。

**取舍理由**：确定性问题本不该让 LLM 去“猜”，交给工具既准又省 token；
LLM 专注它擅长的语义判断。职责分离是降低误报最直接有效的手段。

> **已知局限（诚实说明）**：当前 ruff 检查的是从 diff 中抽取的新增代码片段，
> 而非完整文件，因此对“缩进不完整”“引用了片段外的变量”等情况可能产生少量噪音
> （如 F821 未定义名）。生产化时应改为在完整文件上下文中运行 ruff 并只对变化行报告。

**未来方向**：接入更多语言的 linter 与 SAST（如 bandit 安全扫描），
并用更大的评测集系统调参 precision/recall。

## 五、响应速度

**业界做法**：PR-Agent 的单次调用哲学就是为速度服务。

**我的选择**：默认逐文件分析（对大 PR 稳健），并用上下文预算上限约束单次调用规模，
从而约束延迟与成本上界。代码结构允许未来切换为单次调用模式。

**取舍理由**：逐文件更深入、对超大 PR 更稳；代价是调用次数多。两种模式各有适用场景，
本版优先稳健性，并保留切换空间。

**未来方向**：对独立文件并发调用；对未变化文件结果做缓存。

---

## 六、使用体验

**业界做法**：PR-Agent 提供 `/review` `/describe` `/improve` `/ask` 等斜杠命令。

**我的选择**：Streamlit 网页提供**评审模式**与**问答模式**；问答模式允许对一个真实
PR 自由提问，路演演示效果好。命令行版便于脚本化与 CI 集成。

**未来方向**：做成 GitHub 自动评论机器人，PR 一开就自动评审并回帖。

---

## 七、未来扩展方向（汇总）

1. **多平台**：取数层解耦，新增 `gitlab_client.py` 即可支持 GitLab / Gitee。
2. **自动化**：做成 GitHub App / Action，PR 触发自动评审并回帖。
3. **更强上下文**：引入 AST / 代码依赖图谱。
4. **更多静态工具**：接入 bandit、mypy 等，覆盖安全与类型。
5. **模型路由**：按 PR 复杂度动态选模型。
6. **团队规则**：`review_config.yaml` 已支持自定义关注点，可进一步做成完整规则系统。
