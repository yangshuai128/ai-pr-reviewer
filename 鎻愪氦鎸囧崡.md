## ✅ 提交前必须补的 3 件事（目前的缺口）

### 1. 运行截图 / 演示视频（最重要）
评委不一定会配密钥亲自跑你的项目。**没有运行证据 = 无法证明它真能用。**
- 在 README 顶部加 2–3 张实际运行截图（网页评审结果、命令行报告）。
- 最好再录一段 30–60 秒演示视频，传 B 站/网盘，链接放 README。

### 2. 真实评估数据
`eval/cases.json` 现在是占位示例（假链接）。换成 **3–5 个真实公开 PR**，
人工标注每个 PR 里真实存在的问题，跑 `python eval/run_eval.py`，
把真实的召回率数字写进 `docs/architecture.md`。
> 这是你最强的差异化点，占位数据等于浪费了它。

### 3. 你自己理解并改写代码（关乎诚信红线）
题目明确：**代码重复率超 50% 或抄袭 → 取消资格 + 进黑名单。**
本项目是脚手架/起点，你**必须**：
- 读懂每个模块，能在路演时讲清楚原理；
- 做出你自己的改进（见下方「建议改进点」）；
- 用自己的话重写关键注释和文档。

## 🛠 建议改进点（既提升作品，又降低重复率风险）

挑 2–3 个动手做，让作品带上你的个人印记：
1. **支持单次调用模式**：现在是逐文件调用，加一个“把所有 diff 合并成一次调用”的开关并对比效果。
2. **风险按置信度排序/过滤**：界面加一个滑块，只显示“高置信度”风险，演示误报控制。
3. **导出报告**：把评审结果导出成 Markdown 文件，方便贴进 PR。
4. **支持 GitHub diff 直链或 .diff 文件**：扩展取数来源。
5. **加缓存**：同一 PR 重复分析时直接读缓存，演示响应速度优化。
6. **更友好的错误提示**：把常见错误（无 key、限速、链接错误）翻译成中文友好提示。

## 📤 提交步骤（照做）

### 第一步：本地真实跑通并截图
```bash
# 解压后进入目录
cd ai-pr-reviewer

# 建环境装依赖
python -m venv venv
# Windows: venv\Scripts\activate    Mac/Linux: source venv/bin/activate
pip install -r requirements.txt

# 配密钥：复制 .env.example 为 .env，填入真实 key
# 验证取数
python src/github_client.py
# 跑网页版并截图
streamlit run app.py
```

### 第二步：把真实结果补进项目
- README 加截图；`eval/cases.json` 换真实 PR 并跑出真实数字写进文档；做几个自己的改进。

### 第三步：推送到 GitHub
```bash
cd ai-pr-reviewer
git init
git add .
git commit -m "AI PR Review 助手 - 题目三"
```
去 GitHub 网页新建仓库（**先设 Private**），按提示执行：
```bash
git remote add origin https://github.com/你的用户名/仓库名.git
git branch -M main
git push -u origin main
```
> ⚠️ 推送后立刻去 GitHub 网页确认：**`.env` 没有被上传**（项目已配 `.gitignore`，但务必亲眼确认一次）。

### 第四步：填报名
去 hr.qiniu.com，填「题目三」+ 你的仓库链接。**选题后 24 小时内完成。**

### 第五步：6月1日 00:00 把仓库改公开
GitHub 仓库 → Settings → 最底部 Danger Zone → Change visibility → **Public**。
